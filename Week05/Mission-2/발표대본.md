# Mission-2 발표 대본: Refresh Token 구현

수정한 파일은 `apis/axios.ts` 하나입니다. Mission-1 코드와 비교하면서 설명하겠습니다.

---

## Mission-1과의 차이

**[Mission-1 axios.ts 보여주며]**

Mission-1에서는 요청 인터셉터만 있었습니다. Access Token을 헤더에 붙여주는 역할만 하고, 응답 인터셉터는 없었기 때문에 토큰이 만료돼서 401이 오면 그냥 에러가 났습니다.

---

## Mission-2에서 추가된 것

**[Mission-2 axios.ts 상단 보여주며]**

Mission-2에서 맨 위에 이 두 가지를 추가했습니다.

```typescript
interface CustomInternalAxiosRequestConfig extends InternalAxiosRequestConfig {
  _retry?: boolean;
}

let refreshPromise: Promise<string> | null = null;
```

`_retry`는 무한 루프 방지용 플래그이고, `refreshPromise`는 중복 요청 방지용 전역 변수입니다. 역할은 이따가 설명드리겠습니다.

---

**[refreshAxios 선언 부분 보여주며]**

그리고 axios 인스턴스를 두 개 만들었습니다.

```typescript
export const axiosInstance = axios.create({ ... });
const refreshAxios = axios.create({ ... });
```

`axiosInstance`는 기존처럼 일반 API 요청에 쓰고, `refreshAxios`는 토큰 갱신 전용입니다. 분리한 이유는 뒤에서 설명드리겠습니다.

---

**[응답 인터셉터 전체 보여주며]**

핵심은 여기, 새로 추가한 응답 인터셉터입니다. 401 응답이 왔을 때 세 단계로 처리합니다.

**[`!originalRequest._retry` 조건 가리키며]**

먼저 이 조건입니다. `_retry`가 `false`일 때만 갱신을 시도합니다. 재시도한 요청이 또 401이 나면 `_retry`가 이미 `true`라서 이 블록을 건너뛰고 그냥 에러를 반환합니다. 이게 무한 루프를 막는 장치입니다.

**[`isRefreshRequest` 조건 가리키며]**

다음으로 이 부분입니다. 만약 401이 난 요청이 Refresh 요청 자체라면 바로 로그아웃 처리합니다. Refresh Token도 만료됐다는 뜻이기 때문입니다.

**[`refreshPromise` 블록 가리키며]**

그다음이 실제 갱신 로직입니다. `refreshPromise`가 없을 때만 갱신 요청을 새로 만들고, 이미 진행 중이라면 기존 Promise를 그대로 씁니다. 여러 API가 동시에 401을 받아도 Refresh 요청은 딱 한 번만 나가는 이유가 이 때문입니다.

여기서 갱신 요청을 `refreshAxios`로 보내는 것도 중요합니다. 만약 `axiosInstance`를 쓰면 Refresh 요청도 응답 인터셉터를 타게 되고, 또 401 → 또 갱신 시도하는 무한 루프가 생기기 때문에 인터셉터 없는 별도 인스턴스를 씁니다.

**[마지막 `.then` 부분 가리키며]**

```typescript
return refreshPromise.then((newAccessToken) => {
  originalRequest.headers.Authorization = `Bearer ${newAccessToken}`;
  return axiosInstance.request(originalRequest); // 원래 요청 재시도
});
```

갱신에 성공하면 새 Access Token을 헤더에 넣고, 실패했던 원래 요청을 재시도합니다. 사용자 입장에서는 아무것도 모르고 서비스를 계속 이용할 수 있습니다.
